base模型和chat模型的区别是什么

base模型是在大规模语料上以LM这种形式的预训练任务（预测下一个token）训练得来的；chat模型是在base模型上sft+rlhf上微调得来的。base模型已经有很强大的能力了，但是需要tuning即sft+强化学习rlhf后得到chat模型，对齐了人类的自然语言解锁base模型的强大能力。简而言之，base模型很强但是听不懂人话，需要用少量的tuning数据来让它听懂人话，进而和人类对话。

RLHF

deepspeed

deepspeed的使用有很多功能的实现依赖于配置文件的参数配置，为了弄清楚配置文件中每个参数的意义，本文主要是对官方配置文档的学习使用记录。

一、批量大小的相关参数

1、train_batch_size[int]

有效训练批量大小。这是导致模型更新一步的数据样本量。train_batch_size由单个 GPU 在一次前向/后向传递中处理的批量大小（又称为train_micro_batch_size_per_gpu）、梯度累积步骤（又称为gradient_accumulation_steps）和 GPU 数量共同决定。如果同时提供了train_micro_batch_size_per_gpu和gradient_accumulation_steps，则可以省略。

2、train_micro_batch_size_per_gpu[int]

一个GPU one step 处理的批量大小（无梯度累积）。如果同时提供了train_batch_size和gradient_accumulation_steps，则可以省略

3、gradient_accumulation_steps

在平均和应用梯度之前累积梯度的训练步骤数。此功能有时对于提高可扩展性很有用，因为它会导致步骤之间梯度通信的频率降低。此功能的另一个影响是能够在每个 GPU 上使用更大的批量大小进行训练。如果同时提供了train_batch_size和train_micro_batch_size_per_gpu，则可以省略

DeepSpeed 原生支持Adam、AdamW、OneBitAdam、Lamb和OneBitLamb优化器（可参考：Optimizers - DeepSpeed 0.10.1 documentation）

除此之外，deepspeed的实现中这里还支持其它两个参数：

1）torch_adam：默认为False,代表是否使用torch实现的adam而不使用deepspeed实现的。

2、1-bit Adam optimizer

优化器的介绍参考：1-bit Adam: Up to 5x less communication volume and up to 3.4x faster training

相比于标准的Adam这里额外支持3个参数

1）freeze_step：将1位压缩应用于通信之前的预热步骤数

2）cuda_aware：指示底层 MPI 库支持 CUDA 感知通信

3）comm_backend_name：指示要使用的后端实现

这些都是用于分布式训练时通信相关的参数。

三、通过FP16训练相关的参数

配置使用混合精度进行训练，通过利用 NVIDIA 的 Apex 软件包。

1、enabled

默认为False，标识是否使用混合精度训练

2、auto_cast

默认为False，标识是否把输入强制转换为fp16数据类型

3、loss_scale

表示 FP16训练的损失缩放值。默认值0.0启用动态损失缩放，否则该值将用于静态固定损失缩放

4、initial_scale_power

表示初始动态损失比例值的功率。实际损失规模计算为 2^{initial_scale_power }

5、loss_scale_window

表示升高/降低动态损失比例值的窗口

6、hysteresis

表示动态损耗缩放中的延迟偏移

7、min_loss_scale

表示最小动态损失比例值

五、FP16 训练的 ZeRO 优化

启用和配置ZeRO内存优化的参数：

[]中记录的是可选值

在配置文件中添加zero_optimization的配置才表示启用该优化器

1、stage

选择 ZeRO Optimizer 的不同阶段。阶段0、1、2和3分别指禁用、优化器状态分区、优化器+梯度状态分区和优化器+梯度+参数分区。

2、offload_param

参数卸载：启用和配置参数卸载到 CPU/NVMe 的 ZeRO 优化。仅适用于 ZeRO 阶段 3。请注意，如果未指定或不支持“device”的值，则会触发断言。

3、offload_optimizer

优化器卸载：启用和配置 ZeRO 优化，将优化器计算卸载到 CPU 并将状态卸载到 CPU/NVMe。CPU 卸载适用于 ZeRO 阶段 1、2、3。NVMe 卸载仅适用于 ZeRO 阶段 3。请注意，如果未指定或不支持“device”的值，将触发断言。

5、ZeRO-2 与 ZeRO-3 性能对比

如果其他配置相同，ZeRO-3 可能会比 ZeRO-2 慢，因为除了 ZeRO-2 之外，前者还必须收集模型权重。如果 ZeRO-2 满足您的需求，并且您不需要扩展到几个 GPU 之外，那么您可以选择坚持使用它。重要的是要了解，ZeRO-3 以速度为代价实现了更高的可扩展性。

6、如何选择使用哪个 ZeRO 阶段和卸载以获得最佳性能

一般来说，以下内容适用：

  • 速度方面（左边比右边快）

阶段 0 (DDP) > 阶段 1 > 阶段 2 > 阶段 2 + 卸载 > 阶段 3 > 阶段 3 + 卸载

  • GPU 内存使用情况（右侧的 GPU 内存效率高于左侧）

阶段 0 (DDP) < 阶段 1 < 阶段 2 < 阶段 2 + 卸载 < 阶段 3 < 阶段 3 + 卸载

因此，当您想要在适应最少数量的 GPU 的同时获得最快的执行速度时，您可以遵循以下流程。我们从最快的方法开始，如果遇到 GPU OOM，我们就会转向下一个较慢的方法，但会使用更少的 GPU 内存。等等等等。

首先将批量大小设置为 1（您始终可以使用梯度累积来获得任何所需的有效批量大小）。

启用--gradient_checkpointing 1（HF Trainer）或直接model.gradient_checkpointing_enable()- 如果 OOM 那么

首先尝试 ZeRO 第 2 阶段。如果 OOM 那么

尝试 ZeRO 阶段 2 + offload_optimizer- 如果 OOM 那么

切换到 ZeRO 阶段 3 - 如果 OOM 那么

启用offload_param-cpu如果 OOM 那么

启用offload_optimizer-cpu

一块GPU使用deepspeed的好处：

1）它具有 ZeRO 卸载功能，可以将一些计算和内存委托给主机的 CPU 和 RAM，从而为模型的需求留下更多 GPU 资源 - 例如更大的批量大小，或者能够拟合通常不适合的非常大的模型。

2）它提供了智能 GPU 内存管理系统，可最大限度地减少内存碎片，从而再次允许您适应更大的模型和数据批次。
